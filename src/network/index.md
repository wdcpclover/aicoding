# 互联网架构：从零理解网络世界

> 在写第一行代码之前，你需要先搞懂互联网是怎么工作的。

## 1. 从你每天的操作说起

你每天都在做这些事：

- 打开微信聊天
- 刷抖音看视频
- 用淘宝买东西
- 在浏览器里搜百度

这些操作看起来很简单，但背后其实有一整套系统在运转。今天我们就来拆解这个过程。

---

## 2. 什么是互联网？

互联网，说白了就是 **一堆电脑通过网线（或无线信号）连在一起，互相传数据**。

你的手机是一台电脑，淘宝的机房里也有一堆电脑。当你打开淘宝 App 的时候，你的手机就通过网络去找淘宝的电脑，问它："给我看看首页有什么商品"，淘宝的电脑就把数据传回来，你就看到了页面。

这里有两个角色：

| 角色 | 谁 | 干什么 |
|------|------|------|
| **客户端（Client）** | 你的手机 / 电脑 / 浏览器 | 发出请求，展示结果 |
| **服务器（Server）** | 淘宝 / 微信的机房电脑 | 接收请求，处理数据，返回结果 |

这就是互联网最基本的工作模式：**客户端发请求，服务器给响应**。

---

## 3. 两种经典架构：C/S 和 B/S

互联网应用有两种基本架构，搞懂这两个，后面的内容就好理解了。

### 3.1 C/S 架构（Client/Server，客户端/服务器）

**C/S = 你需要装一个专门的软件（客户端）来使用服务。**

生活中的例子：

- **微信**：你得先下载微信 App（客户端），才能聊天。聊天消息通过网络发到微信的服务器，再转发给对方
- **王者荣耀**：你得下载游戏（客户端），游戏数据存在腾讯的服务器上
- **QQ 音乐**：你得装 App（客户端），歌曲存在服务器上

工作流程：

```
你的手机（安装了微信 App）  ←——网络——→  微信服务器（存消息、好友列表等）
       客户端 Client                         服务器 Server
```

**特点：**
- 需要下载安装专门的 App
- 体验通常更好、更流畅
- 每个平台（安卓、iOS、Windows）可能需要开发不同的版本
- 更新需要用户重新下载

### 3.2 B/S 架构（Browser/Server，浏览器/服务器）

**B/S = 你只需要一个浏览器，不用装任何软件。**

生活中的例子：

- **百度搜索**：打开浏览器，输入 baidu.com，就能搜索
- **12306 买火车票**：打开浏览器就能买，不一定要装 App
- **教务系统**：学校的选课系统，浏览器打开就能用

工作流程：

```
你的浏览器（Chrome / Safari）  ←——网络——→  网站服务器（处理搜索、存数据等）
       浏览器 Browser                         服务器 Server
```

**特点：**
- 不需要安装，打开浏览器就能用
- 天然跨平台——电脑、手机、平板都行
- 更新在服务器端完成，用户无感知
- 体验可能没有原生 App 那么丝滑

### 3.3 对比总结

| 对比项 | C/S 架构 | B/S 架构 |
|--------|---------|---------|
| 是否需要安装 | 需要下载 App | 不需要，浏览器直接打开 |
| 用户体验 | 更流畅 | 依赖浏览器，稍弱 |
| 跨平台 | 每个平台要单独开发 | 天然跨平台 |
| 更新方式 | 用户需要更新 App | 服务器更新，用户无感 |
| 开发成本 | 较高（多端开发） | 较低（一套代码） |
| 典型例子 | 微信、抖音、王者荣耀 | 百度、知乎网页版、教务系统 |

### 3.4 现在的趋势

很多应用其实 **C/S 和 B/S 都做**：

- 淘宝有 App（C/S），也有网页版（B/S）
- 微信有 App（C/S），也有网页版（B/S）
- 甚至微信小程序，本质上也是 B/S 架构——用 Web 技术开发，跑在微信这个"浏览器"里

**我们这门课学的就是 B/S 架构里的 Web 开发。** 因为：
1. 一套代码多端运行
2. 学习成本低
3. AI 工具支持最好

## 4. 当你在浏览器输入网址，发生了什么？

这是面试经典题，也是理解互联网的关键。我们用大白话走一遍。

假设你在浏览器里输入 `www.baidu.com`，然后按回车：

### 第 1 步：找地址（DNS 解析）

互联网上的电脑不认识 "baidu.com" 这种名字，它们只认 **IP 地址**（一串数字，类似门牌号）。

所以浏览器要先去问 **DNS 服务器**（相当于电话簿）：

```
浏览器："baidu.com 的 IP 地址是多少？"
DNS 服务器："是 39.156.66.10"
```

就像你要去一个朋友家，你知道他的名字，但你需要先查到他家的门牌号。

### 第 2 步：建立连接（TCP 三次握手）

知道地址后，浏览器要和百度服务器"打个招呼"，建立连接：

```
浏览器 → 服务器："你好，我想和你通信"   （第一次握手）
服务器 → 浏览器："好的，我准备好了"     （第二次握手）
浏览器 → 服务器："收到，我们开始吧"     （第三次握手）
```

为什么要三次？就像打电话：
- "喂？" —— 确认对方能听到
- "喂，我能听到你" —— 确认你也能听到对方
- "好的，那我们开始说正事" —— 确认双方都没问题

### 第 3 步：发送请求（HTTP 请求）

连接建好后，浏览器发送一个 **HTTP 请求**：

```
GET / HTTP/1.1
Host: www.baidu.com
```

翻译成人话就是：**"请把百度首页的内容给我"**。

### 第 4 步：服务器处理并返回（HTTP 响应）

百度服务器收到请求后，把首页的 HTML、CSS、图片等数据打包，返回给浏览器：

```
HTTP/1.1 200 OK
Content-Type: text/html

<html>
  <head><title>百度一下</title></head>
  <body>...</body>
</html>
```

其中 `200 OK` 表示 **"一切正常，这是你要的数据"**。

### 第 5 步：浏览器渲染页面

浏览器拿到 HTML、CSS、JavaScript 后，把它们 **画（渲染）** 成你看到的网页。

### 完整流程图

```
你输入 www.baidu.com
        ↓
   DNS 解析（查 IP 地址）
        ↓
   TCP 三次握手（建立连接）
        ↓
   发送 HTTP 请求（"给我首页"）
        ↓
   服务器处理请求
        ↓
   返回 HTTP 响应（HTML/CSS/JS）
        ↓
   浏览器渲染页面
        ↓
   你看到了百度首页
```

## 5. HTTP：互联网的通信语言

浏览器和服务器之间传数据，靠的就是 **HTTP 协议**（HyperText Transfer Protocol，超文本传输协议）。

你可以把 HTTP 理解为 **一套对话规则**：客户端怎么提问，服务器怎么回答，双方都按这个规则来。

### 5.1 HTTP 请求（客户端说的话）

每个 HTTP 请求包含四个部分：

| 部分 | 是什么 | 举例 |
|------|--------|------|
| **请求方法** | 你要干什么 | GET（拿数据）、POST（提交数据） |
| **URL** | 你要访问哪里 | `/api/users` |
| **请求头** | 附加信息 | 我是什么浏览器、我接受什么格式的数据 |
| **请求体** | 要发送的数据 | 注册时填的用户名密码（POST 才有） |

### 5.2 HTTP 响应（服务器说的话）

每个 HTTP 响应也包含几部分：

| 部分 | 是什么 | 举例 |
|------|--------|------|
| **状态码** | 处理结果 | 200（成功）、404（找不到）、500（服务器炸了） |
| **响应头** | 附加信息 | 数据类型、缓存策略 |
| **响应体** | 返回的数据 | HTML 页面内容 或 JSON 数据 |

### 5.3 常见状态码

记住这几个就够了：

| 状态码 | 含义 | 大白话 |
|--------|------|--------|
| **200** | OK | 成功，数据给你 |
| **301** | Moved Permanently | 这个地址搬家了，去新地址找 |
| **400** | Bad Request | 你发的请求有问题，我看不懂 |
| **401** | Unauthorized | 你没登录，先登录再来 |
| **403** | Forbidden | 你没权限，不给你看 |
| **404** | Not Found | 你要的东西不存在 |
| **500** | Internal Server Error | 服务器挂了，不是你的问题 |

### 5.4 HTTPS：加了锁的 HTTP

你可能注意到有些网址是 `https://`，多了一个 `s`，这个 `s` 代表 **Secure（安全）**。

- **HTTP**：数据是明文传输的，就像寄明信片，路上谁都能看
- **HTTPS**：数据是加密的，就像寄密封信，只有收件人能看

现在的网站基本都用 HTTPS，尤其是涉及密码、支付的场景。

## 6. 前端和后端

### 6.1 什么是前端？

**前端 = 用户看得到的部分。**

你在手机上看到的界面、按钮、动画、文字、图片——这些都是前端负责的。

前端的三大技术：

| 技术 | 作用 | 打个比方 |
|------|------|----------|
| **HTML** | 定义内容和结构 | 房子的骨架（墙、门、窗） |
| **CSS** | 定义样式和外观 | 房子的装修（颜色、布局、家具） |
| **JavaScript** | 定义交互和行为 | 房子的电器（开关灯、开门、电梯） |

一个简单的例子：

```html
<!-- HTML：结构 -->
<button id="btn">点我</button>
```

```css
/* CSS：样式 */
#btn {
  background-color: green;
  color: white;
  padding: 10px 20px;
  border: none;
  border-radius: 5px;
}
```

```javascript
// JavaScript：交互
document.getElementById('btn').addEventListener('click', function() {
  alert('你点了按钮！')
})
```

### 6.2 什么是后端？

**后端 = 用户看不到的部分。**

当你在淘宝搜索 "手机壳"，前端只是把你输入的关键词发出去。真正去数据库里找商品、排序、筛选的，是后端。

后端做的事情：

- **处理业务逻辑**：比如判断你的密码对不对、你的余额够不够
- **操作数据库**：存数据、查数据、改数据、删数据
- **返回结果**：把处理好的数据发给前端展示

### 6.3 什么是数据库？

**数据库 = 存数据的仓库。**

你注册了一个账号，你的用户名、密码、头像等信息存在哪？存在数据库里。

你可以把数据库想象成一个 **Excel 表格**：

**users 表（用户表）：**

| id | name | email | password |
|----|------|-------|----------|
| 1 | 小明 | xm@qq.com | `(已加密)` |
| 2 | 小红 | xh@qq.com | `(已加密)` |
| 3 | 小刚 | xg@qq.com | `(已加密)` |

后端可以对这个表做四种操作，叫做 **CRUD**：

| 操作 | 英文 | 含义 | 举例 |
|------|------|------|------|
| **C** | Create | 创建 | 新用户注册，往表里加一行 |
| **R** | Read | 读取 | 查看用户信息 |
| **U** | Update | 更新 | 修改密码 |
| **D** | Delete | 删除 | 注销账号 |

### 6.4 前端、后端、数据库怎么协作？

用一个 **用户登录** 的例子来串起来：

```
1. 你在登录页面输入用户名和密码（前端）
          ↓
2. 前端把用户名和密码发送给后端（HTTP 请求）
          ↓
3. 后端收到后，去数据库查：这个用户名存在吗？密码对吗？（后端 + 数据库）
          ↓
4. 数据库返回查询结果给后端
          ↓
5. 后端把结果返回给前端（HTTP 响应）
   - 成功：返回用户信息 + 登录令牌
   - 失败：返回 "用户名或密码错误"
          ↓
6. 前端根据结果显示对应的界面（前端）
   - 成功：跳转到首页
   - 失败：弹出错误提示
```

画成图：

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│          │  请求    │          │  查询    │          │
│   前端   │ ──────→ │   后端   │ ──────→ │  数据库  │
│ (浏览器) │         │ (服务器) │         │ (MySQL等)│
│          │ ←────── │          │ ←────── │          │
│          │  响应    │          │  结果    │          │
└──────────┘         └──────────┘         └──────────┘
   你能看到的            你看不到的           你看不到的
```

## 7. API：前后端沟通的桥梁

### 7.1 什么是 API？

**API（Application Programming Interface）= 应用程序编程接口。**

这个名字很抽象，用大白话说：**API 就是后端提供给前端的一组"服务窗口"。**

你可以把它想象成 **餐厅的菜单**：

- 餐厅（后端）有很多菜可以做
- 菜单（API）告诉你：有哪些菜、怎么点、多少钱
- 你（前端）照着菜单点菜就行，不需要知道厨房怎么做的

### 7.2 API 长什么样？

一个 API 通常由 **请求方法 + URL** 组成。比如一个用户管理系统的 API：

| 功能 | 请求方法 | URL | 说明 |
|------|---------|-----|------|
| 获取所有用户 | GET | `/api/users` | 查看用户列表 |
| 获取单个用户 | GET | `/api/users/1` | 查看 1 号用户的信息 |
| 创建用户 | POST | `/api/users` | 注册新用户 |
| 修改用户 | PUT | `/api/users/1` | 修改 1 号用户的信息 |
| 删除用户 | DELETE | `/api/users/1` | 删除 1 号用户 |

### 7.3 四种请求方法详解

#### GET — 拿数据

<b>"我要看"</b>，不会改变服务器上的任何数据。

```
GET /api/users
```

服务器返回：

```json
[
  { "id": 1, "name": "小明", "email": "xm@qq.com" },
  { "id": 2, "name": "小红", "email": "xh@qq.com" }
]
```

生活类比：去图书馆借书看，书还在图书馆。

#### POST — 提交数据

<b>"我要创建"</b>，向服务器发送新数据。

```
POST /api/users
Content-Type: application/json

{
  "name": "小刚",
  "email": "xg@qq.com",
  "password": "123456"
}
```

服务器返回：

```json
{ "id": 3, "name": "小刚", "message": "注册成功" }
```

生活类比：去银行开户，填表交资料，银行帮你创建账户。

#### PUT — 修改数据

<b>"我要改"</b>，更新服务器上已有的数据。

```
PUT /api/users/3
Content-Type: application/json

{
  "name": "小刚刚",
  "email": "xgg@qq.com"
}
```

生活类比：去银行改手机号，更新你的个人信息。

#### DELETE — 删除数据

<b>"我要删"</b>，从服务器上移除数据。

```
DELETE /api/users/3
```

生活类比：去银行注销账户。

### 7.4 什么是 JSON？

你注意到上面 API 传输的数据格式都是 JSON。

**JSON（JavaScript Object Notation）= 一种通用的数据格式。**

前后端之间传数据，大家都用 JSON，因为它 **简单、易读、所有编程语言都支持**。

JSON 长这样：

```json
{
  "name": "小明",
  "age": 20,
  "hobbies": ["打游戏", "看电影", "写代码"],
  "isStudent": true
}
```

规则很简单：
- 用 `{}` 包裹对象
- 用 `[]` 包裹数组
- 键名用双引号
- 值可以是字符串、数字、布尔值、数组、对象

### 7.5 RESTful API

你可能会听到 **RESTful** 这个词。它不是一个新技术，而是一种 **设计 API 的风格规范**。

核心思想：**用 URL 表示资源，用 HTTP 方法表示操作。**

好的 RESTful 设计：

```
GET    /api/students          → 获取所有学生
GET    /api/students/1        → 获取 1 号学生
POST   /api/students          → 创建新学生
PUT    /api/students/1        → 修改 1 号学生
DELETE /api/students/1        → 删除 1 号学生
```

不好的设计（但也能用）：

```
GET /api/getAllStudents
GET /api/getStudentById?id=1
POST /api/createStudent
POST /api/updateStudent
POST /api/deleteStudent
```

RESTful 的好处是 **看 URL 和方法就知道这个 API 干什么**，不需要额外的文档说明。

## 8. 实际案例：一个选课系统的架构

把前面学的知识串起来，看一个真实的例子——学校选课系统。

### 系统角色

```
学生（前端用户） → 浏览器打开选课页面
教务系统前端     → Vue 3 写的网页
教务系统后端     → Node.js + Express 写的服务
数据库           → 存课程、学生、选课记录
```

### API 设计

| 功能 | 方法 | URL | 谁用 |
|------|------|-----|------|
| 查看所有课程 | GET | `/api/courses` | 学生 |
| 查看课程详情 | GET | `/api/courses/1` | 学生 |
| 选课 | POST | `/api/enrollments` | 学生 |
| 退课 | DELETE | `/api/enrollments/1` | 学生 |
| 查看我的课表 | GET | `/api/my/courses` | 学生 |

### 选课流程

```
学生打开浏览器，进入选课页面
        ↓
前端发送 GET /api/courses，获取课程列表
        ↓
后端从数据库查出所有课程，返回 JSON
        ↓
前端把课程列表渲染成卡片/表格
        ↓
学生点击"选课"按钮
        ↓
前端发送 POST /api/enrollments，带上课程 ID
        ↓
后端检查：这门课还有名额吗？这个学生选过了吗？
        ↓
  ├─ 有名额 → 在数据库里记录选课，返回 200（成功）
  └─ 没名额 → 返回 400（"该课程已满"）
        ↓
前端根据返回结果显示"选课成功"或"课程已满"
```

## 9. 开发者工具：自己动手看请求

学了这么多理论，不如自己看看真实的网络请求。

### 打开浏览器开发者工具

1. 打开 Chrome 浏览器
2. 按 `F12`（Windows）或 `Cmd + Option + I`（Mac）
3. 点击 **Network（网络）** 标签
4. 刷新页面

你会看到一堆请求，每一个就是浏览器和服务器的一次对话。

### 你能看到什么？

点击任意一个请求，可以看到：

- **General**：请求的 URL、方法、状态码
- **Request Headers**：请求头信息
- **Response Headers**：响应头信息
- **Response / Preview**：服务器返回的数据

### 动手试试

1. 打开 `baidu.com`
2. 打开开发者工具的 Network 标签
3. 在搜索框输入 "hello"，点搜索
4. 观察 Network 里新增的请求
5. 找到一个 API 请求，看看它的请求方法、URL、返回数据

这就是前端和后端在实时对话！

| **互联网** | 电脑之间通过网络互相传数据 |
| **C/S 架构** | 需要装 App，客户端 + 服务器 |
| **B/S 架构** | 只需要浏览器，浏览器 + 服务器 |
| **HTTP** | 浏览器和服务器之间的通信规则 |
| **DNS** | 把域名翻译成 IP 地址 |
| **前端** | 用户看得到的界面部分 |
| **后端** | 用户看不到的逻辑和数据处理 |
| **数据库** | 存数据的仓库 |
| **API** | 后端提供给前端的服务窗口 |
| **JSON** | 前后端传数据的通用格式 |
| **RESTful** | 一种设计 API 的规范风格 |
| **CRUD** | 增（Create）查（Read）改（Update）删（Delete） |
