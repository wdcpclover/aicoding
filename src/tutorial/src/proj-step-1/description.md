# 项目蓝图：从业务需求到技术方案 {#proj-step-1}

> 真实工作里**没人**会跟你说"做个 chatbox"。
> 你拿到的是一份**业务需求**——它用业务语言描述"用户能做什么"，剩下的"用什么技术、怎么拆模块"是**你的工作**。
> 这一节走完一遍：**需求 → 详细设计 → 实现**——后面 7 节都按这个套路。

## 🎯 第一步：需求（你拿到了什么）

假设你今天上班，老板（或产品经理）把这份**业务需求文档**扔到你桌上：

```
                ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
                       AI 聊天助手 · 需求文档 v1.0
                ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【背景】
  公司内部各部门都在用 AI，但人手一个 ChatGPT 网页太乱：
  对话散在不同账号、新人没账号、消息查不回来。

【目标】
  做一个内部 AI 助手 Web 应用——任何员工都能上来用，
  对话能保存、能查、能管理。

【一期功能】
  1. 员工能"注册"账号（自己设用户名密码）
  2. 员工能登录；登录后关闭浏览器再回来还认得他
  3. 登录后能跟 AI 一问一答
  4. 一个用户可以同时维护**多个对话**，互相独立
  5. 能删除 / 重命名某个对话
  6. 支持配置用哪个 AI（OpenAI / Claude / DeepSeek...）
  7. AI 的 API Key 由用户自己填，**不能让其他人看到**
  8. 退出登录后，别的电脑用同一账号还能看到对话

【约束】
  ─ 公司内网部署，浏览器访问
  ─ 不一定要双向流式（先做能用的版本）
  ─ 一期不需要图片 / 语音 / 文件附件
```

**这就是真实工作的起点**。下一步**不是**打开 VS Code 写代码——是**坐下来想清楚怎么做**。

## 🔍 第二步：详细设计（为了完成需求，我们怎么做）

这一段是教学的核心——**展示一个工程师从需求到方案的思考过程**。

### 设计 1 — 把需求拆成"用户能做的具体动作"

读一遍需求，把每条业务诉求**翻译成"用户在 UI 上做什么"**。这一步叫**用户故事**：

```
1.  作为新员工，我能点"注册"，输入用户名/密码，建账号
2.  作为已注册员工，我能点"登录"，进入应用
3.  作为已登录员工，我能看到我自己所有的对话列表
4.  作为已登录员工，我能点"新对话"开一个空对话
5.  作为已登录员工，我能在对话里输入一句话，几秒后看到 AI 回复
6.  作为已登录员工，我能点对话的删除/重命名按钮
7.  作为已登录员工，我能打开"设置"，选 AI 类型 + 填 API key
8.  作为已登录员工，关闭浏览器隔天回来还认得我（不用重新登录）
9.  作为已登录员工，我换一台电脑登同一账号，仍能看到我所有的对话
10. 作为已登录员工，我能点"退出登录"
```

> 这一步看似废话，但它是后面所有决策的依据。**漏一条 = 漏一个功能**。

### 设计 2 — 每条用户故事 = 哪里存数据 + 谁负责干活

挨条问自己两个问题：

- **存什么**（这条故事要持久化什么数据？）
- **前端 / 后端各做什么？**（注意"前端做"≠"前端能做"）

| # | 用户故事 | 存什么 | 前端做 | 后端做 |
|---|---|---|---|---|
| 1 | 注册 | username + 密码（哈希） | 表单 + 校验 + 发请求 | 校验、查重、哈希存库、发 token |
| 2 | 登录 | 不新增 | 表单 + 发请求 + 存 token | 验密码、发 token |
| 3 | 看我的对话 | 不新增 | 拉列表 + 显示 | 按用户筛选返回 |
| 4 | 新对话 | 新增一条 conversation | 点击 + 发请求 + 更新 UI | 写入 conversation 表 |
| 5 | 一问一答 | 新增两条 message | 输入 + 发送 + 滚动到底 | 存用户消息 + 调 AI + 存 AI 回复 |
| 6 | 删 / 改对话 | 删 / 改 conversation | 点击 + 发请求 + 更新 UI | DELETE / PATCH 数据库 |
| 7 | 设置 AI + key | 新增/更新 user_settings | 表单 + 提交 | 写入设置；**key 永不返回前端** |
| 8 | 关浏览器还认得 | 不新增 | 启动时校验本地 token | /me 验签 |
| 9 | 换电脑还认得 | (同 8) | 同 8 | 同 8 |
| 10 | 退出 | 不新增 | 扔掉本地 token | 不用做事 |

**这张表是整个项目的"骨架"**。从它能推出下面所有结论。

### 设计 3 — 数据模型（从"存什么"列出表）

把"存什么"列里的所有东西归类——这就是**数据库表设计**：

```
要存：
  ─ 用户的账号                    →  users 表
  ─ 用户的 AI 设置（一人一份）     →  user_settings 表
  ─ 用户的对话（一人多个）         →  conversations 表
  ─ 对话里的消息（一对话多条）     →  messages 表
```

四张表，关系：

```
users  1───1  user_settings    （每个用户一份 AI 配置）
users  1───N  conversations    （一个用户有多个对话）
conversations  1───N  messages （一个对话有多条消息）
```

**详细字段设计 → 留到第 2 节专门讲**（用 SQLite + 外键级联）。

### 设计 4 — 前后端怎么对话（接口设计）

从上面"前端做 / 后端做"里，我们抽出**前后端之间的 HTTP 接口**：

```
POST   /api/auth/register      ┐
POST   /api/auth/login         │  鉴权
GET    /api/auth/me            ┘  ── 第 3 节

GET    /api/conversations           ┐
POST   /api/conversations           │  对话 CRUD
PATCH  /api/conversations/:id       │
DELETE /api/conversations/:id       ┘

GET    /api/conversations/:id/messages    ┐  消息
POST   /api/conversations/:id/messages    ┘

GET    /api/settings                ┐  设置
PUT    /api/settings                ┘

POST   是新增、PATCH 是改局部、DELETE 是删
GET    是查（永远没副作用）
```

这就是 **RESTful 风格**——之前前后端篇讲过，这次只是真实地用上。

### 设计 5 — 关键技术决策（"为什么用 X 不用 Y"）

**Q1：用户密码怎么存？**

- ❌ 明文 ── DB 一泄漏全裸奔
- ❌ MD5 / SHA256 ── 太快，GPU 一秒亿次暴力试
- ✅ **bcrypt** ── 故意慢 + 自动加盐

**Q2：登录后下次访问怎么"记住"是谁？**

- ❌ 服务端 Session ── 要存 session 表，水平扩展要共享
- ✅ **JWT** ── token 自带身份，服务端不存东西

**Q3：API Key 放哪？**

- ❌ 前端 localStorage / bundle ── F12 直接看到
- ❌ 浏览器调 OpenAI ── 跨域 + key 暴露
- ✅ **后端代理** ── key 在服务器，浏览器只调自家后端

**Q4：用什么数据库？**

- ❌ MySQL/Postgres ── 教学要装 Docker，半小时装不完
- ❌ JSON 文件 ── 没并发、没事务
- ✅ **SQLite + better-sqlite3** ── 零依赖、单文件

**Q5：用什么前端框架？**

- ✅ 我们前面学过 Vue 3 + Vue Router + Pinia ── **复用学到的**

整理出来的**最终技术栈**：

```
前端：Vue 3 + Vite + Vue Router + Pinia + axios + marked
后端：Node.js + Express + SQLite + bcrypt + jsonwebtoken + helmet + express-rate-limit
```

### 设计 6 — 进程拓扑（前端和后端怎么"长在一起"）

```
        ┌───────────────────────────┐
        │   浏览器（用户）           │
        │   HTML + Vue 应用          │
        └──────────────┬────────────┘
                       │  HTTP + JSON
                       ▼
        ┌───────────────────────────┐
        │   Vue 前端（端口 5173）    │  ← dev：vite dev server
        │   - 路由 / 组件 / Store    │     prod：vite build 静态文件
        │   - axios 调后端           │
        └──────────────┬────────────┘
                       │  /api/*
                       ▼
        ┌───────────────────────────┐
        │   Express 后端（端口 3002）│
        │   - 鉴权 / CRUD / 调 LLM   │
        └────┬─────────────────┬────┘
             │                  │
             ▼                  ▼
       ┌─────────┐        ┌─────────────┐
       │ SQLite  │        │ OpenAI /    │
       │ 文件    │        │ Claude API   │
       └─────────┘        └─────────────┘
```

**开发态**：前 5173、后 3002，两个独立进程。
**生产态**：前端 build 出来的静态文件交给后端 Express 一并 serve——**同域单端口部署**最省心。

## 💻 第三步：实现（接下来 8 节具体怎么写）

把"详细设计"映射成**学习路线**：

```
本节（① 项目蓝图）── 把上面的设计想清楚，搭好空目录
   ↓
第 2 节 ── 实现"设计 3" 的数据模型（SQLite 4 张表）
   ↓
第 3 节 ── 实现"设计 5 Q1+Q2" 的鉴权（bcrypt + JWT）
   ↓
第 4 节 ── 前端怎么调后端（axios 封装）
   ↓
第 5 节 ── 实现"用户故事 #8 关浏览器还认得"
   ↓
第 6 节 ── 实现"用户故事 #1 #2" 的登录注册页 UI
   ↓
第 7 节 ── 实现"用户故事 #3 #4 #5 #6" 的聊天主界面
   ↓
第 8 节 ── 实现"设计 5 Q3" 的后端代理 + 安全
```

**每节都按 "需求 → 详细设计 → 实现" 的同款套路**。读到第 8 节，整个项目自然就出来了。

### 启动这个项目（5 分钟）

需要**两个终端**，对应前后端两个进程：

```bash
# 终端 1：后端（监听 3002）
cd chatbox/server
npm install
npm start
# 期待输出：🤖 Chatbox 后端已启动: http://localhost:3002
```

```bash
# 终端 2：前端（监听 5173）
cd chatbox/web
npm install
npm run dev
# 期待输出：Local: http://localhost:5173/
```

浏览器打开 [http://localhost:5173/](http://localhost:5173/) → 注册 → 登录 → 发消息。
**默认 mock AI**，不用 API key 也能跑。
