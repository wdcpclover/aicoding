# AICoding · Vibe Coding 导论

> 2026 · 面向 AI 时代的全栈开发课程 · 第一讲

## 一、编程方式正在被重写

### 三年，三级跳

过去三年，软件开发发生了一次范式级的变化，速度超过了之前的三十年：

| 时间 | 阶段 | 写代码的方式 |
|---|---|---|
| 2023 | **智能补全** | 你写一半，AI 猜另一半，像输入法的联想 |
| 2024 | **对话编程** | 把需求用自然语言说清楚，AI 给出完整模块 |
| 2025 至今 | **Agent 工作流** | AI 自己读代码库、改多个文件、跑测试、修报错 |

到了 Agent 阶段，开发者的角色发生了质变：从"逐行书写代码的人"，变成了"定义目标、审查结果的人"。

### 这不是未来，是现状

几个可以自己去验证的事实：

- GitHub 官方统计，新提交代码中由 AI 参与生成的比例逐年攀升，部分语言已过半；
- Claude Code、Cursor、GitHub Copilot 已经是一线互联网公司的标准配置，和当年的 Git 一样属于"默认装备"；
- 国内大厂（阿里、字节、腾讯）都推出了自己的 AI 编程工具：通义灵码、Trae、CodeBuddy；
- 2025 年初，一个新词 **vibe coding** 火遍全球——完全用自然语言对话把应用"聊"出来。

> 💬 **课堂讨论**：你已经用过哪些 AI 工具？用它做过什么？有没有让它帮你写过代码或公式？（请几位同学说说）

### 但有一个事实没有变

> **AI 生成的每一行代码，最终都需要一个看得懂它的人来负责。**

AI 会犯错，而且犯错的方式很隐蔽——它给出的代码永远"看起来很对"。代码上线出了问题，承担后果的是人，不是 AI。

所以编程没有消失，编程的基本单位变了——从"一行代码"变成了"一个意图"。这门课要回答的问题是：**在这样的时代，开发者该掌握什么。**

## 二、什么是 Vibe Coding

### 一个 2025 年才出现的词

**Vibe Coding（氛围编程）** 这个说法由 AI 研究者 Andrej Karpathy 在 2025 年初提出，原话大意是：

> "你只管把感觉（vibe）说出来，看着代码生成、跑起来，对了就留下，不对就让它再改一遍——你几乎不去逐行读代码。"

一句话概括：**用自然语言描述意图，让 AI 写代码，你通过"看运行结果"而不是"读源码"来驱动开发。**

### 它和你以前理解的"编程"差在哪

| | 传统编程 | 智能补全 | **Vibe Coding** |
|---|---|---|---|
| 你输出什么 | 每一行代码 | 半行代码 | **意图和判断** |
| 主要动作 | 写 | 写 + 接受补全 | **描述 → 审查 → 再描述** |
| 看什么判断对错 | 读自己的代码 | 读自己的代码 | **看它跑起来的样子** |
| 速度瓶颈 | 打字和查 API | 打字 | **你能不能把需求说清楚** |

### 一次真实的 Vibe Coding 长什么样

```
你：用 Vue 3 做一个待办清单，能加、能删、能勾选完成，
    完成的划掉，数据存 localStorage 刷新不丢。

AI：（生成一个完整组件）

你：（点开预览，发现勾选没有划线效果）
    勾选完成的项目要加删除线，颜色变灰。

AI：（改了 3 行样式）

你：（再看，满意）好，再加一个"清除已完成"的按钮。
```

整个过程里，你一行代码都没手写，但每一步你都在**做判断**：要什么功能、效果对不对、下一步加什么。这就是 vibe coding 的核心节奏——**描述、运行、审查、再描述**。

### 但 Vibe Coding 不等于"不用懂技术"

这是最大的误区。vibe coding 顺手的时候像魔法，可一旦：

- 它生成的代码**跑不起来**，报一长串错——你得看懂错在哪；
- 功能"看起来对"但**数据存错了 / 有安全漏洞**——你得有能力发现；
- 需求一复杂，AI **越改越乱、来回打转**——你得会拆解、会喊停、会自己接管关键部分。

**懂技术的人 vibe coding 是开飞机，不懂的人是闭着眼睛踩油门。** 这门课要把你训练成前者。

> 💬 **课堂讨论**：如果 AI 写的代码你完全看不懂，但它"能跑"，你敢把它交付给客户吗？为什么？

## 三、新的能力栈

### 一张对照表

| 传统开发的核心能力 | AI 时代的核心能力 |
|---|---|
| 记住语法和 API | **把需求表达成 AI 能精确执行的规格** |
| 手写每一行代码 | **读懂并评审 AI 生成的代码** |
| 调试自己的 bug | **设计验证手段，让系统证明自己是对的** |
| 实现单个功能 | **架构拆解：把大问题拆成 AI 能稳定接住的小问题** |

### 能力一：把需求说清楚

同一个需求，两种问法，结果天差地别：

```
❌ "帮我做一个登录页面"
   → AI 只能瞎猜：什么技术栈？要不要验证码？登录后跳哪？

✅ "用 Vue 3 写一个登录页：手机号 + 密码两个输入框，
   手机号需要校验 11 位格式，点击登录调用 POST /api/login，
   成功后跳转到 /home，失败弹出后端返回的错误信息"
   → AI 一次就能给出可用的代码
```

把模糊的想法翻译成精确的规格——这本身就是一种工程能力，而且恰恰需要你懂技术：不知道"POST"、"校验"、"跳转"这些概念的人，写不出第二种问法。

### 能力二：读懂并评审

AI 是你的"高产但需要盯着的实习生"。它写得快，但你得能看出：

- 这段代码**逻辑**对不对？边界情况（空值、超长输入）处理了吗？
- 有没有**安全**问题？密码是不是明文传了？
- 是不是**过度设计**？三行能解决的事写了三十行？

看不懂代码的人，只能祈祷 AI 是对的。

### 能力三：验证

专业开发者和业余玩家的分水岭：**业余玩家看代码"像不像对的"，专业开发者让代码"证明自己是对的"**——运行它、测试它、喂给它极端输入。这门课所有作业的验收标准都是"跑起来给我看"，而不是"代码交上来"。

### 能力四：拆解

AI 接小任务很稳，接大任务会翻车。"做一个电商网站"它做不好，但拆成"商品列表页 → 购物车状态管理 → 下单接口对接"，每一步它都能高质量完成。**会拆解的人，能让 AI 发挥十倍战斗力。**

### 四种能力的共同地基

注意：这四种能力，**每一项都建立在真实的技术理解之上**——

- 看不懂响应式原理，就判断不了 AI 写的 Vue 组件为什么不更新；
- 不理解 HTTP 协议，就看不出 AI 把 token 放错了位置；
- 没有工程化概念，AI 给的代码再好，也组装不成一个能上线的产品。

**AI 是能力的放大器，不是能力的替代品。** 0 乘以任何倍数还是 0。这门课的全部设计都围绕这个判断展开。

## 四、你的工具箱：AI 编程工具全景

工具很多，但按"AI 介入的深度"分，其实只有三类：

| 类型 | 代表工具 | 适合做什么 |
|---|---|---|
| **补全型** | GitHub Copilot、通义灵码 | 写代码时给行内建议，提速但你仍主导 |
| **对话型** | ChatGPT、Claude、各家网页版 | 问概念、生成片段、解释报错、改 bug |
| **Agent 型** | **Claude Code**、Cursor、Trae、CodeBuddy | 读整个项目、跨文件改、自己跑命令修错 |

Vibe coding 主要发生在 **Agent 型**工具里——它能看到你的整个代码库，自己执行、自己验证。

国内外常见选择：

- **Claude Code**（Anthropic）：命令行里的编程 Agent，本课主力工具之一；
- **Cursor**：最流行的 AI 代码编辑器，VS Code 改造而来；
- **GitHub Copilot**：老牌补全 + 对话，生态最广；
- **通义灵码**（阿里）、**Trae**（字节）、**CodeBuddy**（腾讯）：国内大厂方案，中文体验好、免费档位充足。

> 不用全都学。**挑一个 Agent 型工具用熟**，比每个都浅尝一遍有用得多。本课示例以 Claude Code / Cursor 为主，但换成任何一家，思路完全通用。

## 五、这门课的主线

### 一句话概括

**以一条现代 Web 技术栈为载体，完整走一遍"人机协作开发"的全流程**——从单个页面，到前后端分离的完整应用，再到跨端小程序。

### 为什么选 Web 技术栈

1. **一套代码，全平台覆盖**——浏览器、手机、小程序、桌面端、鸿蒙，Web 技术都能到达；
2. **AI 语料最充足**——Web 是全球代码量最大的领域，AI 对它的生成质量、理解深度都是所有方向里最高的；用 AI 学 Web，等于配了个最强陪练；
3. **零门槛启动**——一个浏览器加一个编辑器就能开始，不用装 Android Studio 那种十几 GB 的庞然大物；
4. **就业纵深**——前端、全栈、跨端开发岗位需求量大，且与 AI 工具链结合最紧密。

### 课程路线图

| 阶段 | 内容 | 产出 |
|---|---|---|
| **基础篇** | 互联网架构、HTML / CSS / JavaScript / TypeScript | 看懂一个网页的全部构成 |
| **框架篇** | Vue 3 响应式、组件化、Vue Router、Pinia | 一个完整的单页应用 |
| **前后端篇** | HTTP 协议、axios、Express 后端、接口联调 | 真实的前后端分离应用 |
| **小程序篇** | 小程序架构、uni-app 跨端开发 | 一套代码多端运行 |
| **项目实战篇** | 全栈项目 Chatbox：从需求到部署 | 一个可演示、可写进简历的作品 |
| **复习篇** | 知识体系串讲 + 实战题演练 | 体系化的知识图谱 |

## 六、第一次 Vibe Coding：先跑起来

别等学完才动手。今天就可以体验一次完整的 vibe coding：

1. **打开一个 Agent 工具**（Claude Code、Cursor、或任意一家国产工具）；
2. **说人话提需求**，比如："用 Vue 3 做一个能换算汇率的小页面，输入人民币金额，选目标货币，实时显示结果"；
3. **看它生成、把它跑起来**——这门课配套的[互动教程](/tutorial/)就内置在线编辑器，改了立刻看到效果；
4. **不满意就继续说**："数字要保留两位小数"、"加一个币种切换的下拉框"；
5. **试着读懂它写了什么**——这一步才是你和"只会踩油门的人"的分水岭。

> 跑通的那一刻你会有点上头——这很正常，那就是 vibe。但记住：**让你脱颖而出的，不是你 vibe 得多快，而是你看得懂、纠得正、能负责。**

## 七、关于 AI 的使用约定

这门课对 AI 的态度只有三条：

1. **全程开放**——作业、练习、项目，AI 随便用，用得好是本事；
2. **以运行为准**——代码能跑、能解释清楚每一段在干什么，才算完成；
3. **评价产出**——衡量标准不是"写了多少行"，而是"做出了什么、能不能讲明白"。

换句话说：这门课考核的不是你和 AI 谁写得多，而是你们俩**合作的上限**。
